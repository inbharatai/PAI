# UnoOne Model Acquisition and Distribution

## Objective

UnoOne users must not depend on Hugging Face, Kaggle, Google AI Edge Gallery or another third-party model catalogue after a release artifact has been approved. Third-party services may be used as an upstream acquisition source during engineering, subject to licence and provenance checks. Production downloads must come from UnoOne-controlled storage and a signed catalogue.

## Non-negotiable rules

1. Do not commit multi-gigabyte model weights to Git.
2. Do not add a production URL before the exact artifact exists.
3. Do not commit a guessed file name, byte size or checksum.
4. Do not mark a downloaded model as device-qualified.
5. Preserve the upstream licence, notices and exact immutable revision.
6. Production catalogues must reject unsigned or incorrectly signed manifests.
7. User prompts, speech, screenshots and form contents are not uploaded to the distribution backend.

## Gemma 4 E2B qualification stages

| Stage | Meaning | Evidence required |
|---|---|---|
| `acquired` | Exact upstream file obtained | upstream repository, immutable revision, file name, licence |
| `integrity-verified` | Local bytes measured | byte size and SHA-256 generated by `qualify_artifact.py` |
| `load-tested` | LiteRT-LM loaded the file | device model, Android version, backend, load log |
| `tool-tested` | Core and PageAgent planning tests passed | committed evaluation summary and zero unknown-tool execution |
| `device-qualified` | Stability/performance gate passed | RAM, latency, temperature, battery and sequential-task results |
| `production-approved` | Legal, integrity, security and device gates approved | signed release record and catalogue entry |

The bundled Android manifest intentionally has an empty Gemma URL, hash and size until these gates produce a real approved record.

## Engineering acquisition flow

```text
approved upstream source
        ↓
manual/CI acquisition with accepted licence
        ↓
immutable revision recorded
        ↓
local artifact scan
        ↓
exact size + SHA-256
        ↓
LiteRT-LM load test
        ↓
phone-tool and PageAgent evaluation
        ↓
device qualification
        ↓
UnoOne-controlled object storage
        ↓
signed production catalogue
```

## Integrity record command

```bash
python scripts/models/qualify_artifact.py \
  /absolute/path/to/model.litertlm \
  --id gemma-4-e2b \
  --version <verified-version> \
  --runtime litertlm \
  --license <verified-license> \
  --provider <upstream-provider> \
  --repository <exact-upstream-repository> \
  --revision <immutable-revision> \
  --upstream-file <exact-upstream-file> \
  --minimum-ram-mb 6144 \
  --recommended-ram-mb 8192 \
  --output artifacts/qualification/gemma-4-e2b.json
```

The RAM values above are provisional product gates, not measured performance claims. They must be updated after the device matrix.

## Catalogue enforcement

Development builds may read the bundled unsigned catalogue with a visible warning. Production builds must require:

- valid Ed25519 catalogue signature;
- exact artifact size;
- exact SHA-256;
- supported app/runtime version;
- approved qualification status;
- licence/notice metadata;
- HTTPS UnoOne-controlled download origin.

A bad signature, empty production checksum, origin mismatch or unsupported runtime must fail closed.

## What the backend does

- Distributes signed APKs, models, language packs and catalogues.
- Provides version and compatibility metadata.
- Supports resumable object downloads.
- Serves licences, notices and release notes.

## What the backend does not do

- It does not run Gemma inference.
- It does not receive user voice, prompts, screenshots or browser form data.
- It does not store passwords, OTPs, CAPTCHA answers or payment information.
- It does not remotely control the device.
